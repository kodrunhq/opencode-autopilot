  <action>
    Create `src/orchestrator/handlers/build.ts` implementing the task-per-dispatch cycle per D-04, D-05, D-06.

    **Core logic flow:**

    1. **Parse state.tasks and state.buildProgress** to determine current position.
    2. **If result provided AND reviewPending:**
       - Parse result for CRITICAL findings (check if result JSON has severity "CRITICAL")
       - If CRITICAL: increment strikeCount, dispatch fix (re-dispatch implementer with fix instructions)
       - If no CRITICAL: clear reviewPending, advance to next wave
    3. **If result provided AND NOT reviewPending:**
       - Mark `state.tasks[buildProgress.currentTask].status = "DONE"` (via spread, immutable)
       - Check if all tasks in current wave are DONE
       - If wave complete: set reviewPending=true, indicate review needed
       - If wave NOT complete: find next pending task, dispatch
    4. **If no result (first call or resume):**
       - Group tasks with `groupByWave(state.tasks)`
       - Find first wave with PENDING tasks
       - If single pending task in wave: `action:"dispatch"` to oc-implementer
       - If multiple pending tasks in wave: `action:"dispatch_multi"` with all tasks (per D-06)
    5. **If all tasks DONE and no review pending:** Return `action:"complete"`

    **Review integration per D-05:** After wave completion, the handler returns a dispatch instruction that tells the orchestrator to call `oc_review` with scope "branch". The handler does NOT call `reviewCore()` directly -- it returns dispatch instructions per the tool-returns-instruction pattern. The CONTEXT.md specific idea about calling reviewCore() directly is better interpreted as: the BUILD handler returns a structured review instruction, and the enhanced orchestrateCore (plan 04) can call reviewCore() directly instead of dispatching oc-reviewer as an agent.

    **State updates:** The handler MUST return the updated state alongside the dispatch result. Add an optional `stateUpdates` field to the return or have orchestrateCore handle state updates based on the dispatch result. Recommended: return `{ ...dispatchResult, _stateUpdates: Partial<PipelineState> }` and let orchestrateCore apply them.

    **Task prompt construction:** Each task dispatch includes:
    - Task ID, title, description from state.tasks
    - Artifact refs to PLAN/tasks.md and ARCHITECT/design.md for context
    - Output path for completion report

    **Edge cases:**
    - Empty tasks array -> return error
    - All tasks already DONE -> return complete
    - Strike count > 3 -> return error (max retries exceeded)

    Keep handler under 100 lines. Extract helper functions: `findCurrentWave()`, `findPendingTasks()`, `buildTaskPrompt()`.
