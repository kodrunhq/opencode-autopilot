**Part A: Memory Architecture Recommendation (11-MEMORY-ARCHITECTURE.md)**

Per D-13, produce a specific technical recommendation for the Phase 15 memory system. This document will be the primary reference when Phase 15 is planned.

**Document structure for 11-MEMORY-ARCHITECTURE.md:**

```markdown
# Memory Architecture Recommendation for Phase 15

## Executive Summary
[1-paragraph recommendation: what to build, what to avoid, key tradeoffs]

## Competitor Memory Pattern Analysis

### claude-mem Approach
- Storage: [SQLite + Chroma specifics]
- Capture: [Observation-based, AI summarization]
- Retrieval: [3-layer progressive disclosure with token costs]
- Injection: [System prompt injection mechanism]
- Strengths: [What works well]
- Weaknesses: [Where it falls short]

### ECC Instinct System
- Extraction: [How patterns are extracted mid-session]
- Confidence scoring: [How quality is gated]
- Evolution: [How instincts become skills]
- Pruning: [How stale patterns are removed]
- Strengths: [What works well]
- Weaknesses: [Where it falls short]

### Our Existing Memory (lesson memory, review memory)
- What we have today
- What it lacks for cross-session persistence

## Recommended Architecture

### Storage Layer
- **Technology:** bun:sqlite (Bun's built-in SQLite)
- **Why not Chroma:** [Bun runtime constraint, native dependency risk]
- **Schema outline:**
  - observations table: id, project_id, type, content, summary, created_at, last_accessed, relevance_score
  - projects table: id, path, name, conventions (JSON), last_updated
  - preferences table: id, key, value, confidence, source_session
- **Full-text search:** SQLite FTS5 virtual table on observations.content and observations.summary
- **Location:** ~/.config/opencode/memory/memory.db

### Capture Layer
- **What to capture:** decisions, patterns, errors, preferences, conventions (NOT raw transcripts)
- **How to capture:** Observation-based -- structured types with timestamps and project context
- **When to capture:** Session lifecycle hooks (session.created, session end)
- **AI summarization:** Compress observations before storage (similar to claude-mem but simpler)

### Retrieval Layer
- **3-layer progressive disclosure** (adapted from claude-mem):
  - Layer 1 (Search): FTS5 keyword search + relevance scoring, returns titles/summaries (~50-100 tokens)
  - Layer 2 (Timeline): Chronological view of observations for a project (~200-500 tokens)
  - Layer 3 (Details): Full observation content on demand (~500-1000 tokens per item)
- **Token budget:** Hard cap of 2000 tokens per session injection
- **Relevance scoring:** Time-weighted (decay) + frequency-weighted (access count) + type-weighted (decisions > patterns > errors)

### Injection Layer
- **Mechanism:** Config hook system prompt injection (we already have this pattern from skill injection)
- **When:** session.created hook
- **What to inject:** Top-N relevant observations for current project, filtered by relevance score and token budget

### Decay / Lifecycle
- **Time-based decay:** Observations lose relevance score over time (configurable, default 90 days half-life)
- **Access-based refresh:** Accessing an observation resets its decay clock
- **Manual pruning:** User can invoke cleanup command
- **Storage cap:** Maximum 10,000 observations per project, oldest pruned first

### Scopes (per Phase 15 requirements)
- **Project-level:** Coding conventions, architecture decisions, codebase patterns (stored per project path)
- **User-level:** Workflow preferences, communication style, tool preferences (stored globally)

## What to Avoid
| Anti-Pattern | Why | Alternative |
|-------------|-----|-------------|
| Chroma/vector DB | Native dependency risk in Bun, complexity | SQLite FTS5 |
| Raw transcript storage | Token-wasteful, 10x storage | Observation-based capture |
| Unbounded growth | Context bomb over time | Decay + storage cap |
| File-based memory (CLAUDE.md) | Less reliable than system prompt injection | Config hook injection |
| Complex embedding models | Runtime dependency, token cost | FTS5 keyword search |

## Open Questions for Phase 15
1. Does bun:sqlite support FTS5? (Must validate in Phase 15 research)
2. Token cost of AI summarization during capture (affects session performance)
3. Optimal decay half-life (needs real-world testing)

## Migration Path from Existing Memory
- Lesson memory (src/pipeline/learning/) -> observations with type "lesson"
- Review memory (src/review/memory.ts) -> observations with type "review_finding"
- Both keep working during migration; new system is additive
```
