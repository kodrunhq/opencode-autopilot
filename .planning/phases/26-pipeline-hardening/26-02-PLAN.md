# Phase 26-02: Deterministic Pipeline Contracts

**Date:** 2026-04-04  
**Status:** Proposed  
**Depends on:** `26-01-PLAN.md`

## Goal

The orchestrator advances phases only from validated, typed contracts (not markdown scraping or heuristic JSON extraction), and identical input traces always produce identical state transitions and outcomes.

## Scope (This Plan)

- Eliminate control-flow dependence on markdown task parsing (`tasks.md`) as source-of-truth.
- Add typed result envelopes and dispatch/result correlation.
- Enforce invariant-checked state transitions.
- Make review ingestion deterministic with typed findings contracts.
- Add deterministic replay/regression tests and migration diagnostics.

## Non-Goals (Follow-up)

- Full event-sourced architecture rewrite.
- Replacing all human-readable markdown artifacts.
- Large BUILD handler refactor beyond deterministic attribution requirements.

## Invariants to Enforce (Must Hold After Every State Write)

1. `currentPhase === null` iff pipeline status is terminal (`COMPLETED` or non-recoverable `FAILED`).
2. Exactly one phase is `IN_PROGRESS` when `currentPhase !== null`, and it matches `currentPhase`.
3. `buildProgress.currentTask` is either `null` or references an existing BUILD task.
4. `reviewPending === true` only when `currentPhase === "BUILD"`.
5. Every accepted result envelope matches an existing pending dispatch entry.
6. A `(runId, dispatchId, resultId)` tuple is processed at most once.
7. State revision increases monotonically on every successful mutation.
