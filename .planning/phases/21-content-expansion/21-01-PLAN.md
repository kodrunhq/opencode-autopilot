**1a. Expand coding-standards/SKILL.md (per D-01, D-02, D-03):**

Append two new sections after the existing Section 10 (Code Comments). Keep all existing content unchanged. Add:

**Section 11: OOP Principles (SOLID)**
Cover all 5 SOLID principles with DO/DON'T examples in the same opinionated tone as existing sections:
- **Single Responsibility Principle (SRP):** One class = one reason to change. DO: separate `UserValidator` from `UserRepository`. DON'T: `UserManager` that validates, persists, AND sends emails.
- **Open/Closed Principle (OCP):** Open for extension, closed for modification. DO: use strategy pattern or polymorphism. DON'T: modify existing switch/if chains for each new variant.
- **Liskov Substitution Principle (LSP):** Subtypes must be substitutable for base types. DO: subclass contracts must honor parent contracts. DON'T: `Square extends Rectangle` where setWidth breaks area expectations.
- **Interface Segregation Principle (ISP):** Many specific interfaces over one general-purpose interface. DO: `Readable`, `Writable` separately. DON'T: `IDoEverything` with 20 methods.
- **Dependency Inversion Principle (DIP):** Depend on abstractions, not concretions. DO: constructor injection of interfaces. DON'T: `new DatabaseService()` inside business logic.

Each principle: 1-paragraph explanation + DO code example + DON'T code example.

**Section 12: Composition and Architecture**
Cover composition over inheritance, dependency injection, and Clean Architecture layers:
- **Composition over Inheritance:** DO: compose behaviors via delegation and interfaces. DON'T: deep inheritance hierarchies (>2 levels). Show a practical delegation example vs a fragile inheritance chain.
- **Dependency Injection:** DO: pass dependencies through constructors. DON'T: reach into global singletons or static service locators. Show constructor injection pattern.
- **Clean Architecture Layers:** DO: organize as Domain (entities + business rules) -> Application (use cases) -> Infrastructure (DB, APIs, frameworks). Dependencies point inward. DON'T: let infrastructure leak into domain logic.

This requires changing the return to use a mutable `Set` internally:

```typescript
export async function detectProjectStackTags(projectRoot: string): Promise<readonly string[]> {
  const tags = new Set<string>();

  const results = await Promise.all(
    Object.entries(MANIFEST_TAGS).map(async ([manifest, manifestTags]) => {
      try {
        await access(join(projectRoot, manifest));
        return [...manifestTags];
      } catch {
        return [];
      }
    }),
  );

  for (const result of results) {
    for (const tag of result) {
      tags.add(tag);
    }
  }

  // Check glob-based manifests (file extension matching)
  try {
    const { readdir: readdirFn } = await import("node:fs/promises");
    const entries = await readdirFn(projectRoot);
    for (const [ext, extTags] of Object.entries(EXT_MANIFEST_TAGS)) {
      if (entries.some((entry) => entry.endsWith(ext))) {
        for (const tag of extTags) {
          tags.add(tag);
        }
      }
    }
  } catch {
    // readdir failure is non-fatal
  }

  return [...tags];
}
```

Note: `readdir` is already imported at the top of the file — no, it is NOT currently imported. Add `readdir` to the existing `import { access } from "node:fs/promises"` import statement.
