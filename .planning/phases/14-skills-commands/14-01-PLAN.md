2. **The Brainstorming Process** (inspired by superpowers' Socratic design refinement, 131k+ stars):
   - **Phase 1: Clarify the Problem** — Ask 3-5 Socratic questions to understand the real problem. Questions should challenge assumptions. Example questions: "What problem does this solve for the user?", "What happens if we don't build this?", "What's the simplest version that delivers value?"
   - **Phase 2: Divergent Exploration** — Generate 3-5 distinct approaches. Each approach gets: name, 1-sentence description, key tradeoff, effort estimate (small/medium/large). Do NOT evaluate yet — quantity over quality in this phase.
   - **Phase 3: Convergent Evaluation** — For each approach, assess: feasibility (can we build it with current stack?), alignment (does it solve the clarified problem?), risk (what could go wrong?). Score each dimension LOW/MEDIUM/HIGH.
   - **Phase 4: Synthesis** — Recommend top approach with rationale. If approaches can be combined, propose a hybrid. Document rejected approaches and why (decision log fodder).
   - **Phase 5: Action Items** — Convert chosen approach into concrete next steps: files to create, APIs to design, tests to write.
3. **Socratic Question Templates** — Categorized by domain: requirements ("Who is the user?", "What does success look like?"), architecture ("What are the scaling constraints?", "Where are the boundaries?"), risk ("What's the worst failure mode?", "What do we not know?")
4. **Anti-Pattern Catalog:**
   - Anti-Pattern: Premature Convergence — jumping to a solution before exploring alternatives. Instead: force yourself to generate at least 3 approaches.
   - Anti-Pattern: Analysis Paralysis — exploring endlessly without converging. Instead: time-box divergent phase, then force a decision.
   - Anti-Pattern: Group Think — all approaches look the same. Instead: deliberately propose a "wild card" approach that breaks assumptions.
   - Anti-Pattern: Scope Creep — brainstorming expands beyond the original problem. Instead: revisit the clarified problem statement after each phase.
5. **Integration with Our Tools** — After Phase 5, use the plan-writing skill to decompose into tasks. Reference `oc_orchestrate` for autonomous execution of the chosen approach.
6. **Failure Modes** — When brainstorming produces poor results: unclear problem definition (go back to Phase 1), all approaches are variations of the same idea (add constraints to force creativity), no approach feels feasible (reduce scope or split the problem).

**File 1: `assets/skills/tdd-workflow/SKILL.md`** (target: 300-400 lines)

YAML frontmatter:
```yaml
---
name: tdd-workflow
description: Strict RED-GREEN-REFACTOR TDD methodology with anti-pattern catalog and explicit failure modes
stacks: []
requires: []
---
```

Content sections:
1. **When to Use** — Business logic with defined I/O, API endpoints with request/response contracts, data transformations, validation rules, algorithms, state machines. When NOT to use: UI layout/styling, configuration files, one-off scripts, simple CRUD with no business logic.
2. **The RED-GREEN-REFACTOR Cycle:**
   - **Phase 1: RED (Write a Failing Test)** — Write ONE test that describes the expected behavior. The test MUST fail before writing any production code. Run the test and confirm the failure message matches expectations. If the test passes without implementation, the test is wrong. Commit: `test: add failing test for [feature]`
   - **Phase 2: GREEN (Make It Pass)** — Write the MINIMUM code to make the test pass. Do NOT add features the test doesn't require. Do NOT optimize. Do NOT refactor. The goal is a passing test, nothing more. Resist the urge to write "clean" code — that's Phase 3. Run test — MUST pass. Commit: `feat: implement [feature]`
   - **Phase 3: REFACTOR (Clean Up)** — Now improve the code: extract functions, rename variables, remove duplication. The test MUST still pass after every change. If the test fails during refactor, revert the last change. Commit: `refactor: clean up [feature]`
3. **Test Writing Guidelines** — Test names describe behavior, not implementation (`"rejects expired tokens"` not `"test validateToken"`). One assertion per test (or tightly related assertions). Arrange-Act-Assert structure. Use descriptive failure messages.
4. **Anti-Pattern Catalog (per D-05):**
   - Anti-Pattern: Writing Tests After Code — tests become assertions of implementation, not behavior. Instead: always write the test FIRST.
   - Anti-Pattern: Skipping RED — writing test and implementation together means you don't know if the test actually catches failures. Instead: run the test, see it fail, then implement.
   - Anti-Pattern: Over-Engineering in GREEN — adding error handling, edge cases, optimizations before the test requires them. Instead: write only what the current test needs.
   - Anti-Pattern: Skipping REFACTOR — accumulating technical debt by never cleaning up. Instead: always do a refactor pass, even if it's "looks fine."
   - Anti-Pattern: Testing Implementation Details — mocking internal methods, asserting call counts. Instead: test public API behavior and outputs.
   - Anti-Pattern: Large Test Suites with No RED — all tests written at once, all passing. This isn't TDD. Instead: one test at a time, one cycle at a time.
5. **Integration with Our Tools** — After GREEN phase, invoke `oc_review` for quick quality check on the implementation. Use `oc_forensics` if a test fails unexpectedly during REFACTOR to diagnose root cause.
6. **Failure Modes** — Test won't fail (RED): the behavior is already implemented or the test is wrong. Test won't pass (GREEN): re-read the test, check for typos, simplify the implementation. Refactor breaks tests: revert last change, make smaller refactor steps.

**File 2: `assets/skills/systematic-debugging/SKILL.md`** (target: 250-350 lines)

YAML frontmatter:

```yaml
---
name: systematic-debugging
description: 4-phase root cause analysis methodology for systematic bug diagnosis and resolution
stacks: []
requires: []
---
```

Content sections:
1. **When to Use** — Bug reports, test failures, unexpected behavior, performance regressions, integration issues. When NOT to use: feature requests, code review (use code-review skill), known issues with documented fixes.
2. **The 4-Phase Debugging Process:**
   - **Phase 1: Reproduce** — Confirm the bug exists. Get exact reproduction steps. Create a minimal reproduction case. If you can't reproduce it, gather more context (logs, environment, input data). A bug you can't reproduce is a bug you can't fix.
   - **Phase 2: Isolate** — Narrow the scope. Use binary search: comment out half the code, does the bug persist? Check recent changes (`git log --oneline -20`, `git diff HEAD~5`). Add strategic logging at module boundaries. Identify the exact function/line where behavior diverges from expectation.
   - **Phase 3: Diagnose** — Understand WHY the bug exists, not just WHERE. Read the code path end-to-end. Check assumptions: types, null checks, async timing, state mutations. Look for common root causes: off-by-one errors, race conditions, stale closures, implicit type coercion, missing error handling.
   - **Phase 4: Fix** — Apply the minimal fix. Write a regression test FIRST (TDD-style: test should fail with the bug, pass with the fix). Verify the fix doesn't break other tests. Check for similar patterns elsewhere in the codebase (the same bug often exists in multiple places).
3. **Common Root Cause Patterns** — Race conditions (async operations completing in unexpected order), State mutation (object modified in place when caller expected immutability), Boundary errors (off-by-one, empty array, null/undefined), Type coercion (implicit conversions producing unexpected values), Stale closures (capturing old values in callbacks), Missing error handling (errors swallowed silently).
4. **Anti-Pattern Catalog:**
   - Anti-Pattern: Shotgun Debugging — making random changes hoping something fixes it. Instead: follow the 4-phase process.
   - Anti-Pattern: Fixing Symptoms — adding a null check without understanding why the value is null. Instead: trace to the root cause.
   - Anti-Pattern: No Regression Test — fixing the bug without a test that catches it. Instead: always write the regression test first (Phase 4).
   - Anti-Pattern: Debugging in Production — adding console.log to production code. Instead: reproduce locally first (Phase 1).
5. **Integration with Our Tools** — Use `oc_forensics` to analyze failed pipeline runs (identifies failing phase, agent, root cause). Use `oc_review` after applying the fix to check for introduced issues. Use `oc_logs` to inspect session event history for timing-related bugs.
6. **Failure Modes** — Can't reproduce: ask for exact environment details, check for environment-specific behavior. Reproduce but can't isolate: add more granular logging, check async operation ordering. Root cause unclear: pair with another approach (rubber duck debugging, explain the code to an imaginary colleague).
