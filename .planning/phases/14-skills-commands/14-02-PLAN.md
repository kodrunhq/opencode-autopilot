YAML frontmatter:

```yaml
name: verification
description: Pre-completion verification checklist methodology to catch issues before marking work as done
stacks: []
requires: []
---
```

Content sections:
1. **When to Use** — Before marking any task, feature, or PR as complete. Before committing code. Before merging branches. Before deploying. Any time you're about to say "this is done."
2. **The Verification Checklist** (inspired by superpowers' lightweight pre-completion checklist):
   - **Step 1: Requirements Check** — Re-read the original requirement/task description. Does the implementation address every stated requirement? List each requirement and mark it satisfied or not. Missing requirements = not done.
   - **Step 2: Code Quality Check** — Run linter (`bun run lint` or equivalent). Run type checker (`bunx tsc --noEmit` or equivalent). Check for console.log/debugger statements. Check for TODO/FIXME/HACK comments that should be resolved. Check file sizes (<800 lines per CLAUDE.md).
   - **Step 3: Test Verification** — Run the test suite (`bun test`). Check that new code has test coverage. Check that existing tests still pass. If no tests exist for new functionality, write them before proceeding (reference tdd-workflow skill).
   - **Step 4: Integration Check** — Does the change work with the rest of the system? Check imports — are all new exports consumed correctly? Check types — do interfaces match between producer and consumer? Run the application and verify the feature works end-to-end.
   - **Step 5: Edge Case Review** — What happens with empty input? What happens with very large input? What happens with null/undefined? What happens with concurrent access? What happens with network failures?
   - **Step 6: Security Scan** — No hardcoded secrets (API keys, passwords, tokens). All user inputs validated. Error messages don't leak sensitive data. Authentication/authorization verified on new endpoints.
3. **Integration with Our Tools** — Invoke `oc_review` for automated code review before marking complete. Use `oc_doctor` to verify plugin health. Use `oc_session_stats` to check for error patterns in the current session.
4. **Anti-Pattern Catalog:**
   - Anti-Pattern: "Works on My Machine" — only testing in one environment. Instead: check for environment-specific assumptions.
   - Anti-Pattern: Skipping Tests — "it's a small change." Instead: small changes cause big bugs. Always verify.
   - Anti-Pattern: Reviewing Your Own Code — you'll miss the same things you missed writing it. Instead: use `oc_review` for an independent check.
5. **Failure Modes** — Linter fails: fix issues before proceeding. Tests fail: diagnose with systematic-debugging skill. Type errors: trace the type mismatch to the source.

**File 2: `assets/skills/git-worktrees/SKILL.md`** (target: 200-300 lines)

YAML frontmatter:
```yaml
---
name: git-worktrees
description: Git worktrees for isolated parallel development — work on multiple branches simultaneously without stashing
stacks: []
requires: []
---
```

Content sections:
1. **When to Use** — Working on multiple features simultaneously. Need to switch context without stashing work. Running long-running tests on one branch while developing on another. Reviewing a PR while keeping your current work intact. Comparing implementations side-by-side.
2. **Git Worktrees Workflow:**
   - **Setup: Create a Worktree** — `git worktree add ../project-feature-name feature-branch` creates a new checkout in a sibling directory. Each worktree is a full working copy sharing the same `.git` repository. Changes in one worktree don't affect others.
   - **Naming Convention** — Use `../project-branchname` pattern. Keep worktrees as siblings of the main repo. Example: `../myapp-fix-auth`, `../myapp-feature-search`.
   - **Working in Worktrees** — Each worktree has its own index, HEAD, and working directory. You can run different branches simultaneously. `cd ../project-feature-name` to switch context. Run tests, builds, servers independently in each worktree.
   - **Cleanup** — `git worktree remove ../project-feature-name` when done. `git worktree prune` to clean up stale worktrees. `git worktree list` to see all active worktrees.
3. **Common Patterns:**
   - **Pattern: Parallel Feature Development** — Main worktree: current feature. Second worktree: hotfix branch. Third worktree: PR review. All running simultaneously.
   - **Pattern: Safe Experimentation** — Create a worktree for experimental changes. If the experiment fails, just remove the worktree. No need to reset, stash, or undo anything.
   - **Pattern: Comparison Testing** — Two worktrees: one with the change, one without. Run benchmarks or tests in both to measure impact.
4. **Anti-Pattern Catalog:**
   - Anti-Pattern: Too Many Worktrees — having 10+ worktrees creates confusion. Instead: limit to 2-3 active worktrees.
   - Anti-Pattern: Forgetting to Clean Up — stale worktrees waste disk space. Instead: `git worktree list` regularly, remove when done.
   - Anti-Pattern: Shared Node Modules — worktrees share the git repo but NOT node_modules. Instead: run `bun install` in each worktree.
5. **Failure Modes** — Branch already checked out: each branch can only be checked out in one worktree. Worktree directory exists: remove or choose a different path. Merge conflicts when syncing: handle like normal git conflicts.

**File 1: `assets/skills/plan-writing/SKILL.md`** (target: 250-350 lines)

YAML frontmatter:
```yaml
---
name: plan-writing
description: Methodology for decomposing features into bite-sized implementation tasks with file paths, dependencies, and verification criteria
stacks: []
requires: []
---
```

Content sections:
1. **When to Use** — New feature implementation, refactoring existing code, multi-step bug fixes, any work touching more than 3 files.
2. **The Plan Writing Process** (inspired by superpowers' bite-sized tasks with file paths):
   - **Step 1: Define the Goal** — State what must be TRUE when this work is complete. Outcome-shaped, not task-shaped. Good: "Users can log in with email/password." Bad: "Build the auth system."
   - **Step 2: List Required Artifacts** — For each goal, list the concrete files that must exist: source files, test files, configuration files, types/interfaces. Use exact file paths (e.g., `src/auth/login.ts`, not "the login module").
   - **Step 3: Map Dependencies** — For each artifact, what must exist before it can be built? Draw a dependency graph. Items with no dependencies go first. Items that depend on others come later.
   - **Step 4: Group into Tasks** — Each task: 1-3 files, 15-60 minutes of work, single concern. Each task has: name (action-oriented), files (exact paths), action (specific instructions), verification (how to prove it works), done criteria (measurable).
   - **Step 5: Assign Waves** — Tasks with no dependencies = Wave 1 (can run in parallel). Tasks depending on Wave 1 = Wave 2. And so on. More waves of smaller tasks > fewer waves of larger tasks.
   - **Step 6: Add Verification** — Each task needs a verification command (test, build, lint). The plan as a whole needs an end-to-end verification step.
3. **Task Sizing Guide:**
   - Too small (<15 min): Combine with a related task.
   - Right size (15-60 min): Touches 1-3 files, single concern, clear done criteria.
   - Too large (>60 min): Split by file, by concern, or by layer (types first, then implementation, then tests).
4. **Anti-Pattern Catalog:**
   - Anti-Pattern: Vague Tasks — "Set up the database." Instead: "Add User and Project models to schema.prisma with UUID ids, email unique constraint."
   - Anti-Pattern: No File Paths — "Create the auth module." Instead: "Create src/auth/login.ts with POST handler accepting {email, password}."
   - Anti-Pattern: Horizontal Layers — "Create all models, then all APIs, then all UIs." Instead: vertical slices — "User feature (model + API + UI), then Product feature."
   - Anti-Pattern: Missing Verification — tasks without a way to prove they're done. Instead: every task has a test command or check.
5. **Integration with Our Tools** — Use `oc_orchestrate` to execute the plan automatically. Use `oc_plan` to track task completion status. Use the plan-executing skill for batch execution workflow.
6. **Failure Modes** — Plan too large: split into multiple plans of 2-3 tasks each. Circular dependencies: rethink the architecture, extract shared types into a separate task. Tasks keep growing: you're combining concerns — split them.

**File 2: `assets/skills/plan-executing/SKILL.md`** (target: 200-300 lines)

YAML frontmatter:
```yaml
---
name: plan-executing
description: Batch execution methodology for implementing plans with verification checkpoints after each task
stacks: []
requires:
  - plan-writing
---
```

Content sections:
1. **When to Use** — After writing a plan (using plan-writing skill). When implementing a multi-task feature. When running through a task list systematically.
2. **The Execution Process** (inspired by superpowers' batch execution with verification):
   - **Step 1: Read the Full Plan** — Before implementing anything, read every task. Understand the dependency graph. Identify the critical path. Note any tasks that can run in parallel.
   - **Step 2: Execute Wave by Wave** — Start with Wave 1 tasks (no dependencies). Complete each task fully before moving to the next. After each task: run its verification command, check its done criteria, commit the work.
   - **Step 3: Verify After Each Task** — Run the task's specific verification (test, build, lint). Run the full test suite to catch regressions. If verification fails, fix before proceeding — do NOT continue with a broken base.
   - **Step 4: Track Progress** — Mark completed tasks. Log any deviations from the plan (scope changes, unexpected issues, new tasks discovered). Update time estimates for remaining tasks based on actual velocity.
   - **Step 5: Handle Failures** — Task verification fails: use systematic-debugging skill. Unexpected dependency: update the plan, re-order tasks. Scope creep: log the new requirement, defer to a follow-up plan.
   - **Step 6: Final Verification** — After all tasks complete, run the plan's overall verification. Check all done criteria. Run end-to-end tests. This is the "ship it" gate.
3. **Commit Strategy** — One commit per task. Commit message references the task: `feat(auth): implement login endpoint (task 2/5)`. Each commit should leave the codebase in a working state. Never commit broken code.
4. **Anti-Pattern Catalog:**
   - Anti-Pattern: Skipping Verification — "I'll test it all at the end." Instead: verify after every task.
   - Anti-Pattern: Continuing on Failures — task 2 fails but you start task 3 anyway. Instead: fix task 2 first.
   - Anti-Pattern: Not Committing — doing 5 tasks in one giant commit. Instead: commit after each verified task.
   - Anti-Pattern: Deviating Without Logging — changing the plan without recording why. Instead: log every deviation.
5. **Integration with Our Tools** — Use `oc_orchestrate` or `oc_quick` for autonomous plan execution. Use `oc_review` after each task for quality verification. Use `oc_state` to track pipeline state. Use `oc_phase` to check phase transitions.
6. **Failure Modes** — All tasks fail: the plan may be fundamentally flawed — go back to plan-writing. Velocity is too slow: tasks are too large — split them. Tests pass but feature doesn't work: missing integration test — add one.
