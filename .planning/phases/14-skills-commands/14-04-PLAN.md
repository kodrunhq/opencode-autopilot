Content sections (per D-06: patterns + testing + idioms with framework-specific guidance):
1. **Type-Level Patterns:**
   - Prefer `interface` over `type` for object shapes (better error messages, extendable)
   - Use discriminated unions for state machines: `type State = { status: "idle" } | { status: "loading" } | { status: "error"; error: Error }`
   - Use `readonly` arrays and properties by default. Only remove `readonly` when mutation is explicitly needed.
   - Use `as const` for literal types and frozen objects
   - Avoid `any` — use `unknown` and narrow with type guards. If `any` is unavoidable, add a `// biome-ignore lint/suspicious/noExplicitAny: [reason]` comment
   - Use branded types for domain identifiers: `type UserId = string & { readonly __brand: "UserId" }`
   - Template literal types for string patterns: `type Route = \`/api/${string}\``
2. **Bun Runtime Patterns:**
   - Use `bun test` for testing (built-in, fast, Jest-compatible API)
   - Use `node:fs/promises` for file I/O (not `Bun.file()`/`Bun.write()` per project constraints)
   - Use `Bun.serve()` for HTTP servers (not Express)
   - Import from `bun:sqlite` for SQLite (not a third-party driver)
   - Use `bun:test` matchers: `expect().toBe()`, `expect().toEqual()`, `expect().toThrow()`
   - Test structure: `describe("Module", () => { test("behavior description", () => { ... }) })`
3. **Error Handling:**
   - Return result types instead of throwing: `{ success: true; data: T } | { success: false; error: string }`
   - Catch at boundaries, not in business logic
   - Use `isEnoentError()` pattern for filesystem errors (check error code, not message)
   - Never catch without re-throwing or logging — silent catch is a bug
4. **Module Patterns:**
   - Export pure functions and interfaces, not classes (unless state encapsulation is needed)
   - Use barrel files (`index.ts`) only for public API surfaces
   - Dependency flow: strictly top-down, no cycles
   - File size: 200-400 lines target, 800 max
5. **Testing Patterns:**
   - Use `describe`/`test` (not `it`) for consistency
   - Test pure functions: input -> output, no mocks needed
   - Test side effects: mock the boundary (filesystem, network), verify the call
   - Use `beforeEach` for test isolation, `afterEach` for cleanup
   - Snapshot tests: only for complex output where manual assertion is impractical
6. **Immutability Patterns:**
   - Build objects with spread: `{ ...existing, newProp: value }`
   - Use `Object.freeze()` for constants and configuration objects
   - Use `ReadonlyArray<T>` and `Readonly<Record<K, V>>` for function parameters
   - Never push to arrays — use spread: `[...existing, newItem]`

Content sections:
1. **Error Handling:**
   - Always check errors: `if err != nil { return fmt.Errorf("context: %w", err) }`
   - Wrap errors with `%w` for unwrapping: `fmt.Errorf("failed to open file: %w", err)`
   - Use sentinel errors for expected conditions: `var ErrNotFound = errors.New("not found")`
   - Create custom error types for rich context: `type ValidationError struct { Field, Message string }`
   - Never ignore errors with `_` unless there's a comment explaining why
2. **Concurrency Patterns:**
   - Use channels for communication, mutexes for state protection
   - Always use `context.Context` for cancellation and timeouts
   - Use `errgroup.Group` for parallel tasks with error collection
   - Goroutine lifecycle: every goroutine must have a clear shutdown path
   - Never start a goroutine without a way to stop it
3. **Interface Design:**
   - Keep interfaces small: 1-3 methods. Prefer `io.Reader` over `FileReaderWriterCloser`
   - Accept interfaces, return structs
   - Define interfaces at the consumer, not the provider
   - Use the implicit satisfaction — no `implements` keyword needed
4. **Testing Patterns:**
   - Table-driven tests: `tests := []struct{ name string; input; want }{ ... }`
   - Use `t.Helper()` in test helpers for better error locations
   - Use `t.Parallel()` for independent tests
   - Use `testdata/` directory for test fixtures
   - Use `_test.go` suffix and `package foo_test` for black-box testing
5. **Package Organization:**
   - Flat packages: avoid deep nesting (`pkg/auth/`, not `pkg/auth/handlers/middleware/`)
   - Internal packages (`internal/`) for implementation details
   - `cmd/` for entry points, `pkg/` for library code
   - One package per concern
6. **Anti-Pattern Catalog:**
   - Anti-Pattern: Naked Returns — `return` without values in named-return functions. Instead: be explicit.
   - Anti-Pattern: Interface Pollution — defining an interface before you need it. Instead: extract when you have 2+ implementations.
   - Anti-Pattern: Global State — package-level `var db *sql.DB`. Instead: pass dependencies via function parameters or struct fields.

---
name: python-patterns
description: Pythonic patterns covering type hints, error handling, async, testing with pytest, and project organization
stacks:
  - python
requires: []
---
```

Content sections:
1. **Type Hints:**
   - Use type hints on all function signatures: `def fetch_user(user_id: str) -> User | None:`
   - Use `from __future__ import annotations` for forward references
   - Use `TypedDict` for structured dictionaries, `dataclass` for value objects
   - Use `Protocol` for structural subtyping (duck typing with type checking)
   - Avoid `Any` — use `object` or `Unknown` patterns
2. **Error Handling:**
   - Use specific exceptions: `raise ValueError("age must be positive")` not `raise Exception("bad")`
   - Create custom exceptions inheriting from domain base: `class AuthError(AppError): pass`
   - Use context managers for resource cleanup: `with open(path) as f:`
   - Never bare `except:` — always catch specific exceptions or `except Exception:`
3. **Async Patterns:**
   - Use `async`/`await` for I/O-bound operations (network, file, database)
   - Use `asyncio.gather()` for concurrent tasks
   - Use `asyncio.TaskGroup` (Python 3.11+) for structured concurrency
   - Never mix sync and async — pick one pattern per module
4. **Testing with pytest:**
   - Use `pytest` with fixtures for test setup: `@pytest.fixture`
   - Parametrize tests: `@pytest.mark.parametrize("input,expected", [...])`
   - Use `conftest.py` for shared fixtures
   - Use `pytest.raises(ValueError)` for exception testing
   - Use `tmp_path` fixture for temporary file tests
5. **Project Organization:**
   - `src/` layout with `pyproject.toml`
   - Use `__init__.py` for public API exports only
   - Separate concerns: `models/`, `services/`, `api/`, `utils/`
   - Use `pydantic` for data validation at boundaries
6. **Anti-Pattern Catalog:**
   - Anti-Pattern: Mutable Default Arguments — `def f(items=[])` shares the list across calls. Instead: `def f(items=None): items = items or []`
   - Anti-Pattern: Bare Except — `except:` catches everything including KeyboardInterrupt. Instead: `except Exception:`
   - Anti-Pattern: Star Imports — `from module import *` pollutes namespace. Instead: import explicitly.

stacks:

requires: []
---
```

Content sections:
1. **Ownership and Borrowing:**
   - Prefer borrowing (`&T`, `&mut T`) over taking ownership
   - Use `Clone` only when shared ownership is genuinely needed
   - Prefer `&str` over `String` in function parameters
   - Use lifetimes explicitly when the compiler can't infer them
   - Move semantics: understand when values are moved vs. copied (Copy trait)
2. **Error Handling:**
   - Use `Result<T, E>` for recoverable errors, `panic!` only for unrecoverable bugs
   - Use `?` operator for error propagation
   - Create custom error types with `thiserror`: `#[derive(Debug, thiserror::Error)]`
   - Use `anyhow::Result` in application code, custom errors in library code
   - Pattern match on errors: `match result { Ok(v) => ..., Err(e) => ... }`
3. **Option Patterns:**
   - Use `Option<T>` instead of null/sentinel values
   - Prefer combinators: `.map()`, `.and_then()`, `.unwrap_or_default()`
   - Only use `.unwrap()` in tests or when the invariant is proven
   - Use `if let Some(v) = option` for conditional extraction
4. **Unsafe Guidelines:**
   - Minimize `unsafe` blocks — wrap unsafe code in safe abstractions
   - Document every `unsafe` block with a `// SAFETY:` comment explaining the invariant
   - Never use `unsafe` for performance optimization without benchmarks proving it's necessary
   - Prefer safe alternatives: `Vec` over raw pointers, `Arc<Mutex<T>>` over shared mutable state
5. **Testing:**
   - Unit tests in the same file: `#[cfg(test)] mod tests { ... }`
   - Integration tests in `tests/` directory
   - Use `#[should_panic]` for expected panics
   - Use `proptest` or `quickcheck` for property-based testing
   - Use `assert_eq!` with descriptive messages: `assert_eq!(result, expected, "failed for input: {input}")`
6. **Anti-Pattern Catalog:**
   - Anti-Pattern: Unnecessary Clone — cloning to satisfy the borrow checker without thinking. Instead: restructure code to use references.
   - Anti-Pattern: Unwrap Everywhere — `.unwrap()` in production code. Instead: use `?` or handle the error.
   - Anti-Pattern: Stringly Typed — using `String` for everything. Instead: use newtypes and enums for domain concepts.
