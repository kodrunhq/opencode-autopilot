**File 1: `assets/skills/code-review/SKILL.md`** (target: 200-300 lines)

YAML frontmatter:
```yaml
---
name: code-review
description: Structured methodology for requesting and receiving code reviews — what to check, how to provide feedback, and how to respond to review comments
stacks: []
requires:
  - coding-standards
---
```

Content sections:
1. **When to Use** — Before merging any PR. After completing a feature. When reviewing someone else's code. When `oc_review` flags issues that need human judgment.
2. **Requesting a Review:**
   - Provide context: what the change does, why it's needed, what alternatives were considered
   - Highlight risky areas: "I'm unsure about the error handling in auth.ts lines 40-60"
   - Keep PRs small: <300 lines of meaningful diff (exclude generated files)
   - Self-review first: use the verification skill before requesting review
3. **Performing a Review** — Review in this order for maximum value:
   - **Architecture:** Does the overall approach make sense? Are responsibilities properly separated?
   - **Correctness:** Does the code do what it claims? Edge cases handled? Error paths covered?
   - **Security:** Input validation? Auth checks? No leaked secrets? (Reference coding-standards skill)
   - **Performance:** N+1 queries? Unbounded loops? Missing indexes? Unnecessary allocations?
   - **Readability:** Clear naming? Small functions? Reasonable file sizes?
   - **Testing:** Tests exist for new behavior? Existing tests still pass? Edge cases tested?
4. **Providing Feedback** — Use severity levels: CRITICAL (must fix), HIGH (should fix), MEDIUM (consider), LOW (nit). Be specific: file, line, what's wrong, suggested fix. Be constructive: explain WHY something is a problem, not just WHAT.
5. **Responding to Review Comments** — Address every comment (fix or explain why not). Don't take feedback personally. Ask for clarification if a comment is unclear. Mark resolved comments after addressing them.
6. **Integration with Our Tools** — Use `oc_review` for automated multi-agent review. Use coding-standards skill as the baseline for quality checks. The review engine runs 13 specialist agents in parallel covering logic, security, quality, testing, and more.
7. **Anti-Pattern Catalog:**
   - Anti-Pattern: Rubber-stamp Reviews — approving without reading. Instead: spend at least 10 minutes per 100 lines.
   - Anti-Pattern: Style-Only Reviews — only commenting on formatting. Instead: focus on correctness and architecture first.
   - Anti-Pattern: Blocking on Nits — holding up a PR for trivial style issues. Instead: approve with suggestions for LOW items.

**File 2: `assets/skills/strategic-compaction/SKILL.md`** (target: 200-250 lines)

YAML frontmatter:
```yaml
---
name: strategic-compaction
description: Context window management through strategic summarization — keep working memory lean without losing critical information
stacks: []
requires: []
---
```

Content sections:
1. **When to Use** — Context window is filling up (>60% used). Working on a long task spanning many files. Need to switch subtasks without losing prior context. Session is slowing down or responses are becoming less detailed.
2. **The Compaction Process** (inspired by ECC's strategic-compact skill):
   - **Step 1: Identify What to Keep** — Current task requirements (active goal). Key decisions made so far (and why). File paths and function signatures currently being modified. Error messages or test failures being investigated. Constraints and invariants that must be maintained.
   - **Step 2: Identify What to Compact** — File contents already read and understood (just keep the path + key insight). Exploration dead ends (just keep "tried X, didn't work because Y"). Verbose tool output (keep the conclusion, discard raw output). Background context not needed for the current subtask.
   - **Step 3: Create a Summary** — Structure: "Working on [goal]. Key decisions: [list]. Current state: [what's done, what's next]. Active files: [paths]. Constraints: [list]." Target: 500-1000 tokens for the summary. This summary becomes the "restore point" if context needs to be rebuilt.
   - **Step 4: Apply Compaction** — In a new session or after context reset, load the summary first. Then read only the files actively being modified. Re-read constraints and interfaces only if needed.
3. **What to NEVER Compact** — Active error messages being debugged. Type definitions/interfaces being implemented against. Test expectations being satisfied. Security constraints and validation rules.
4. **Anti-Pattern Catalog:**
   - Anti-Pattern: Compacting Too Early — summarizing before you've explored enough. Instead: compact when context is >60% full.
   - Anti-Pattern: Losing Key Decisions — summarizing away the "why" behind a decision. Instead: always keep decision rationale.
   - Anti-Pattern: Over-Compacting — reducing to so little that you lose important context. Instead: keep file paths, function signatures, and decision rationale.
5. **Failure Modes** — Summary is too vague: add specific file paths and function names. Lost critical context: re-read the relevant files (they're on disk, context is recoverable). Summary is too long: focus on what's needed for the NEXT task, not everything done so far.

**File 3: `assets/skills/e2e-testing/SKILL.md`** (target: 200-250 lines)

YAML frontmatter:
```yaml
---
name: e2e-testing
description: End-to-end testing patterns for critical user flows — test the system as a user would use it
stacks: []
requires: []
---
```

Content sections:
1. **When to Use** — Critical user flows (login, checkout, data creation). Integration points between multiple services. Flows where unit tests can't catch the issue (timing, routing, middleware). Before major releases.
2. **E2E Test Design Principles:**
   - Test user journeys, not individual components. A login test: navigate to login page -> fill email -> fill password -> click submit -> verify redirect to dashboard.
   - Use realistic data, not "test123". Use data that exercises edge cases.
   - Keep tests independent — each test should work regardless of test execution order.
   - Use page object pattern (or equivalent) to encapsulate UI interactions.
3. **Test Structure:**
   - **Arrange:** Set up test data, navigate to starting page, ensure clean state.
   - **Act:** Perform the user actions (click, type, navigate).
   - **Assert:** Verify the expected outcome (page content, URL, API response).
   - **Cleanup:** Remove test data, reset state (if needed).
4. **Common E2E Patterns:**
   - **Pattern: Happy Path First** — Test the expected successful flow before testing error cases.
   - **Pattern: Smoke Tests** — A minimal set of E2E tests that verify the application starts and basic flows work. Run these on every commit.
   - **Pattern: Critical Path Tests** — Tests for the most important user flows (sign up, core feature, payment). Run these before every release.
5. **Anti-Pattern Catalog:**
   - Anti-Pattern: Testing Everything E2E — E2E tests are slow. Instead: use unit tests for logic, integration tests for APIs, E2E only for critical flows.
   - Anti-Pattern: Flaky Tests — tests that pass/fail randomly. Instead: avoid timing dependencies, use explicit waits, ensure clean state.
   - Anti-Pattern: No Cleanup — tests leaving data behind that breaks other tests. Instead: clean up after each test.
6. **Integration with Our Tools** — Use `oc_review` to check E2E test quality. Reference tdd-workflow skill for the RED-GREEN-REFACTOR approach when writing E2E tests.
7. **Failure Modes** — Test is flaky: add explicit waits, check for race conditions. Test passes locally but fails in CI: check for environment differences (ports, timeouts, screen size). Test suite is too slow: reduce E2E scope, move non-critical tests to integration level.
