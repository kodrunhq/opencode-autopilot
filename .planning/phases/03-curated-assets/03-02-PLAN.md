    The file must have YAML frontmatter with two fields:
    - name: coding-standards
    - description: Universal coding standards and best practices for code review and generation. Covers naming, file organization, error handling, immutability, and separation of concerns.

    Body content uses opinionated, actionable rules (per D-12: clear "DO this, DON'T do that"). Language-agnostic principles (per D-11).

    The skill MUST cover these 10 areas:

    1. Naming Conventions - descriptive names, consistent casing, booleans as questions, functions as verbs. DON'T use single letters, abbreviations, generic names.
    2. File Organization - under 400 lines (800 max), one module = one concern, organize by feature/domain. DON'T mix unrelated functionality.
    3. Function Design - under 50 lines, single responsibility, max 3-4 nesting levels, limit parameters to 3. Return early for guard clauses.
    4. Error Handling - handle explicitly, never swallow errors, user-friendly messages in UI, detailed context server-side. DON'T use exceptions for control flow.
    5. Immutability - new objects over mutation, spread/map/filter/reduce, DON'T reassign params or use push/splice on shared state. Exception: mutation OK when API requires it.
    6. Separation of Concerns - data access vs business logic vs presentation, interfaces at boundaries. DON'T mix HTTP with business rules.
    7. DRY - extract when duplicated 3+ times, DON'T over-abstract (YAGNI), prefer composition over inheritance.
    8. Input Validation - validate at system boundaries, schema-based validation, DON'T trust external data. Fail fast.
    9. Constants and Configuration - named constants, config files, DON'T hardcode URLs/keys/timeouts.
    10. Code Comments - comment WHY not WHAT, DON'T leave commented-out code, JSDoc for public APIs.

    Include concrete code examples (language-agnostic pseudocode) for each area. Use markdown headers, bullet lists, and code blocks. At least 100 lines.
  </action>
